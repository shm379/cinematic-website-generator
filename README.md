# 🎬 Cinemate — سازنده‌ی وب‌سایت سینمایی با هوش مصنوعی

یک پلتفرمِ کامل: کاربر می‌آید، **ثبت‌نام می‌کند**، حوزه‌ی کاری‌اش را می‌گوید و در یک **جریانِ onboarding صفر تا صد** یک **وب‌سایت سینمایی تک‌صفحه‌ایِ اسکرول‌محور** می‌سازد — خروجی را **دانلود** می‌کند یا **روی هاست ما منتشر** می‌کند.

> الهام‌گرفته از طراحیِ لندینگِ SaaSهای مدرن (مثل docsbot.ai)، با تمِ سینماییِ تیره. ساخته‌شده با **GSAP + ScrollTrigger + Lenis**. بدون React، بدون Build Step. خروجی هر سایت **یک فایل HTML خودکفا**.

---

## ✨ چه چیزی ساخته شده؟

* **لندینگِ کامل (`/`)** — هیرو با **پیش‌نمایش زنده‌ی قابل‌تعویض**، بخش ویژگی‌ها، «چطور کار می‌کند»، **نمونه‌های زنده‌ی قابل‌کلیک**، قیمت‌گذاری/میزبانی، FAQ و CTA.
* **ثبت‌نام / ورود** — مودالِ تمیز (شبیه‌سازی‌شده‌ی کلاینت‌ساید؛ امکانِ «ادامه به‌عنوان مهمان»).
* **ویزاردِ onboarding ۶ مرحله‌ای** — حوزه → برند → ظاهر (رنگ/موتیف) → محتوا → بازبینی (دسکتاپ/موبایل) → **تحویل و میزبانی**، همه با **پیش‌نمایش زنده‌ی کناری**.
* **تحویلِ خروجی** — دانلود HTML، کپی کد، باز کردن در تب جدید.
* **پرسشِ میزبانی** — «دوست داری روی هاست ما منتشر شود؟» با پلن‌ها و یک لینکِ زنده (ساب‌دامین/دامنه‌ی اختصاصی).
* **ویرایشگرِ پیشرفته (`/builder`)** — پنلِ کاملِ تنظیمات برای کاربرانِ حرفه‌ای.
* **موتورِ پارامتریک (`/generator.js`)** — هیروی **پروسیجرالِ Canvas** (بدون نیاز به ویدیو)، ۱۰+ پریستِ حوزه، دوزبانه (FA/EN).

## 🚀 اجرا (لوکال)

```bash
npm install
node server.js
```

| مسیر | توضیح |
|------|-------|
| `/` | **لندینگ + ثبت‌نام + ویزاردِ onboarding** |
| `/builder` | ویرایشگرِ پیشرفته |
| `/gallery` | **گالریِ تصاویر** — جست‌وجوی عکسِ استوک از Pexels |
| `/demo` | نمونه‌ی واقعی و پولیش‌شده («خشخاش») |
| `/healthz` | health check (برای Docker/Coolify) |
| `POST /api/generate` | تولیدِ سمت‌سرور؛ بدنه = config → `{ html }` |
| `GET /api/site?field=coffee&brand=Cafe` | رندرِ مستقیمِ یک سایتِ تولیدشده |
| `POST /api/generate-from-prompt` | **مگاپرامپت**: `{ prompt }` → `{ via, config, html }` |
| `POST /api/image` | **عکسِ یک کارت**: عکسِ استوکِ Pexels یا تولید با AI → `{ url, source }` |
| `POST /api/photos` | **عکسِ گروهیِ کارت‌ها**: `{ field, count }` → `{ photos: [{ url, … }], source }` (پرکردنِ خودکارِ کارت‌ها) |
| `GET /api/gallery?query=coffee&page=1` | **جست‌وجوی گالری** در Pexels → `{ photos, next_page, total_results }` |

> همه‌ی منطقِ ساختِ سایت **کلاینت‌ساید** است؛ پس روی هاستِ استاتیک (Vercel) هم کار می‌کند. اندپوینت‌های `/api/*` (و قابلیت‌های AI) نیازمندِ سرور هستند.

## 🤖 قابلیت‌های هوش مصنوعی (اختیاری)

* **مگاپرامپت → سایت** (`POST /api/generate-from-prompt`): یک توصیفِ متنیِ آزاد بده، یک کانفیگِ کامل + HTML بگیر. ترتیبِ انتخاب: اگر `NABU_BASE_URL` تنظیم شده باشد از **NabuGate** (دروازهٔ مرکزیِ هوش مصنوعی، سازگار با OpenAI) استفاده می‌شود تا پروژه مستقیم به هیچ providerی وصل نشود؛ وگرنه اگر `ANTHROPIC_API_KEY` باشد با **Claude** تحلیل می‌شود؛ در غیر این‌صورت یک **پارسرِ بدونِ کلید** به‌کار می‌رود — پس همیشه کار می‌کند. در ویزارد، جعبه‌ی «✨ بساز با پرامپت» در قدم ۱.
* **عکسِ کارت‌ها** (`POST /api/image` و `POST /api/photos`): وقتی حوزه‌ای انتخاب می‌شود، ویزارد به‌صورت خودکار یک دستهٔ عکسِ استوک برای کارت‌ها می‌گیرد (`/api/photos` — یک جست‌وجو، چند عکسِ متمایز) تا سایت از همان ابتدا عکس داشته باشد؛ با دکمه‌ی «🎨 عکسِ دیگر» روی هر کارت هم می‌شود عکسِ همان کارت را تک‌به‌تک عوض کرد (`/api/image`). منبعِ عکس‌ها همان زنجیرهٔ استوک است (پروکسیِ NabuGate → Pexels)؛ اگر `OPENAI_API_KEY` باشد، `/api/image` در صورتِ نبودِ عکسِ استوک با `gpt-image-1` عکسِ اختصاصی می‌سازد و در `public/generated/` ذخیره می‌کند. بدونِ هیچ منبعی، کارت‌ها طرحِ پیش‌فرض (گرادیان + حرفِ اول) را نگه می‌دارند.
* **گالریِ تصاویر** (صفحهٔ `/gallery` + `GET /api/gallery`): یک صفحهٔ جداگانه برای **جست‌وجوی عکسِ استوک از Pexels** — با گریدِ ماسونری، صفحه‌بندی («بارگذاری بیشتر»)، و ذکرِ منبع/عکاس (طبقِ شرایطِ Pexels). عکس‌ها ترجیحاً از **پروکسیِ عکسِ NabuGate** می‌آیند (`{NABU_BASE_URL}/v1/photos/search` — کلیدِ Pexels فقط داخلِ دروازه می‌ماند)؛ بدونِ NabuGate، با `PEXELS_API_KEY` مستقیم به Pexels وصل می‌شود. بدونِ هیچ‌کدام، صفحه یک پیامِ راهنما نشان می‌دهد. بدونِ کوئری، فیدِ «منتخب»ِ Pexels نمایش داده می‌شود.

> 🎬 **تیمِ ساخت (ساب‌ایجنت‌ها):** هفت متخصصِ این سبک (کارگردانِ خلاق، طراحِ تعاملی،
> موشن، سه‌بعدی، فرانت‌اند، محتوا، کارایی/دسترس‌پذیری) به‌صورتِ ساب‌ایجنت‌های NabuGate
> تعریف شده‌اند و با یک درخواست اجرا می‌شوند. جزئیات در [`AGENTS.md`](AGENTS.md).

متغیرهای محیطی (همه اختیاری) — به `.env.example` نگاه کن:

```bash
NABU_BASE_URL=https://nabugate.internal  # مسیرِ دروازه (بدونِ /v1)؛ مسیرِ ترجیحیِ AI
NABU_API_KEY=nabu_...                     # کلیدِ پروژه در NabuGate
NABU_MODEL=nabu-smart                     # هر alias یا ایجنتِ NabuGate (مثلِ cine-content-strategist)
PEXELS_API_KEY=...              # حالتِ مستقیم/لوکال؛ با NABU_BASE_URL لازم نیست (کلید در خودِ دروازه است)
OPENAI_API_KEY=sk-...           # ساخت عکس با AI (فالبک)
OPENAI_IMAGE_MODEL=gpt-image-1  # پیش‌فرض
ANTHROPIC_API_KEY=sk-ant-...    # مگاپرامپت وقتی NabuGate تنظیم نشده
CLAUDE_MODEL=claude-opus-4-8    # پیش‌فرض
```

> `@anthropic-ai/sdk` یک **وابستگیِ اختیاری** است؛ اگر نصب نشود یا کلید نباشد، مگاپرامپت به پارسرِ بدونِ کلید برمی‌گردد.

## 🐳 استقرار با Docker / Coolify

### Docker (اجرای محلی)
```bash
docker build -t cinemate .
docker run -p 3000:3000 cinemate          # → http://localhost:3000
```
> با `docker compose` برای دسترسیِ محلی، بلوکِ `ports:` در `docker-compose.yml` را از کامنت دربیاور (در Coolify باید کامنت بماند). سلامتِ کانتینر از طریقِ `/healthz` بررسی می‌شود.

### Coolify
1. در Coolify یک **Resource** جدید از روی این مخزنِ Git بساز.
2. Build Pack را روی **Docker Compose** (یا **Dockerfile**) بگذار.
3. **Deploy** — Coolify دامنه، SSL و reverse-proxy را خودش مدیریت می‌کند.

**چرا ترافیک حالا می‌رسد؟** سه چیز هماهنگ شده تا Coolify بفهمد کجا را پروکسی کند:
- `server.js` به `process.env.PORT` روی `0.0.0.0` گوش می‌دهد و اگر پورت توسطِ پلتفرم تنظیم شده باشد، **دقیقاً** به همان bind می‌شود (دیگر به پورتِ آزادِ بعدی نمی‌پرد).
- `docker-compose.yml` متغیرِ جادوییِ Coolify یعنی **`SERVICE_FQDN_WEB_3000`** را دارد؛ این به Coolify می‌گوید یک دامنه بسازد و پروکسی‌اش را به پورت **3000** سرویسِ `web` وصل کند.
- `Dockerfile` با `EXPOSE 3000` و `HEALTHCHECK` روی `/healthz` همان پورت را اعلام می‌کند.

> برای دامنه‌ی اختصاصی، در تنظیماتِ Coolify مقدارِ `SERVICE_FQDN_WEB_3000=https://your-domain.com` را بگذار (یا خالی بگذار تا خودکار ساخته شود).

> همین زیرساخت، پشتِ گزینه‌ی «میزبانی روی هاست ما» در ویزارد قرار می‌گیرد: هر سایتِ تولیدشده یک فایلِ استاتیک است که Coolify می‌تواند روی ساب‌دامین/دامنه منتشرش کند.

## 🛠 استفاده‌ی برنامه‌نویسی از موتور

`public/generator.js` یک ماژول **UMD** است (مرورگر + Node):

```js
const CWG = require('./public/generator.js');
const html = CWG.generate({
  brand: 'خشخاش', field: 'tea', lang: 'fa', accent: '#f59e0b'
});
// html = یک فایلِ کاملِ خودکفا. هر فیلدِ نداده‌شده از پریستِ حوزه پر می‌شود.
```

حوزه‌های آماده: `tea, coffee, perfume, jewelry, realestate, restaurant, fitness, clinic, tech, fashion` و `_default`.

## 🤖 نگه‌داریِ خودکار با Claude (GitHub Actions)

این مخزن دو workflow دارد که نگه‌داری را خودکار می‌کنند:

* **بررسیِ روزانه‌ی باگ** — هر روز کد را اسکن می‌کند، issueهای با برچسبِ `auto-fix` را رفع می‌کند و یک **Pull Request پیش‌نویس** می‌سازد (اگر باگی نباشد، PRای ساخته نمی‌شود).
* **`@claude` در issueها/کامنت‌ها** — هرجا `@claude` بنویسید، همان لحظه بررسی/اصلاح کرده و در صورت نیاز PR می‌سازد.

راه‌اندازی فقط یک قدم دارد: Secret به نامِ `CLAUDE_CODE_OAUTH_TOKEN` (توکنِ اشتراکِ Claude) را اضافه کنید. جزئیاتِ کامل در [`.github/AUTOMATION.md`](.github/AUTOMATION.md).
